PROJECT RULES

GENERAL ENGINEERING PREFLIGHT GATE

Before every response, action, code edit, build, commit, explanation, or verification:

1. Classify the next output: analysis, proposal, code edit, build, commit, explanation, or verification.
2. Do not modify code or behavior unless the user has explicitly authorized implementation after the strategy is clear.
3. Identify the source of truth and do not invent missing data.
4. Prefer fixing the upstream data structure, lifecycle, ownership, or algorithm over applying an output correction.
5. Reject hard-coded values, ad-hoc heuristics, hidden state, and silent fallbacks unless explicitly approved as the intended design.
6. Choose the simplest responsibility-correct solution; avoid accumulating workaround layers.
7. If source data is missing, inconsistent, or ambiguous, report it clearly instead of silently compensating.
8. Before claiming completion, run the relevant checks or state exactly what could not be verified.

FAST ADMIN PATH

For non-development administrative tasks, such as updating release metadata, removing temporary release files, publishing gh-pages manifests, or simple Git housekeeping:

1. Apply the project preflight, but only for scope, authorization, safety, and source-of-truth checks.
2. Do not run runtime tests, Spectral, architectural analysis, or broad regression checks unless the change can affect application behavior, generated OAS content, templates, or runtime configuration.
3. Perform only the minimal file checks needed to avoid touching unintended files.
4. Verify with git diff/status and, if pushed, a final remote/tree/status check.
5. Keep user updates short and avoid exploratory detours.
6. If the administrative task reveals a possible behavior or data issue, stop treating it as Fast Admin Path and switch back to the full engineering preflight.

BRANCH POLICY

1. master is the production branch and must always represent the latest consolidated and accepted production state.
2. Development work must happen on non-master development branches.
3. After a development branch has been merged into master and the merge has been verified, that branch is considered absorbed and should be deleted locally and remotely unless the user explicitly asks to keep it.
4. Do not use old absorbed branches as functional documentation. Preserve relevant history through master commits, tags, release notes, or project documentation instead.
5. gh-pages is a separate publishing branch for release metadata and update manifests. Do not treat it as an application development branch and do not merge it into master.
6. Ask the user before merging any development branch into master or deleting remote branches.

ONE-SHOT INPUT MAINTENANCE PATH

For explicitly requested one-shot activities, fast-track activities, or tasks that only update/manipulate external input artifacts, templates, Excel files, generated deliverables, reference data, or other non-application assets:

1. Treat the task as one-shot input maintenance, not application development.
2. Do not create or switch Git branches unless explicitly requested.
3. Do not stage, commit, or version helper scripts, scratch reports, or generated verification artifacts unless explicitly requested.
4. Put temporary manipulation scripts, reports, generated OAS, logs, and comparison outputs outside the repository, preferably under C:\tmp.
5. If editing files outside Git/workspace protection, create a recoverable backup before any in-place edit.
6. Use only the declared source-of-truth files for data changes.
7. Run focused artifact verification: import/open checks, targeted content checks, generated output checks when relevant, and Spectral when OAS is generated.
8. Report changed files, backup path, verification evidence, and residual issues.
9. If the work reveals a need to change OASIS application code, runtime behavior, generators, converters, tests, or configuration, stop the one-shot path and ask whether to switch to the normal development workflow.

1. The user prompting requests is the project master. In case of doubt, before exploring possibilities, ask directly for information. In many cases he can give you the information you need.
Don't take long sequences of actions without interacting with him. Sometimes he wishes to give hints, recommendations, help, but you are too selfishly busy in finding a solution. Stop frequenty and ask.

2. Don't code anything before asking to the user. Writing it in tne implementation plan (which by policy is automatically approved) is not enough. Ask directly for the authorization to proceed.

3. All information included in OAS files must be derived by the Excel templates. Any kind of heuristics is strictly forbidden. Hard coding data is strictly forbidden. Taking information (e.g. values, enumerations) from sources that are not the Excel templates is stricltly forbidden.

4. The application must produce deterministic results with must depend exclusively on what has been written in the Excel templates. And must be able to work with different schemas, parameters, responses, bodies, etc. No ad-hoc solution targeted to address an issue are acceptable.

5. After any progress, evaluate the results against functional and quality regressions. Perform a QA assessment and produce a report with mandatory (if any) and recommended actions.

6. Key files, to understand the quality achieved, are the OAS present in the "Expected results" folder.

7. Git, git, git. Remember to version every change. Use snapshots before changes and use always development branches (create a new development branch at each session). The master branch must always contain the last consolidated and accepted result. Ask always to the user if a branch can be merged to master.

8. All reverts must be requested and authorized. Temporary reverts for tests anche checks can be executed autonomously only after having stashed the current situation. Nothing must be lost for a bad git practice.

9. Artifacts (implementation_plan.md, task.md, etc.) must be kept up-to-date as the "single source of truth" for the project state and plan.

10. Excel files are the ONLY source of truth for OAS data. If data is missing or erroneous in Excel, DO NOT hardcode fixes. Instead, generate a specific report highlighting the missing/incorrect data so it can be corrected in the source.

11. Always use Spectral linter to check if there are errors to fix and produce a report to the user with some action suggested.

12. Quality Assurance: every implementation you make must be reviewed acting as an experienced software architect. Meke sure that all state of the art architecural and design patterns are followed and that the code is maintainable, scalable, and secure. Strictly avoid all ad-hoc solutions and antipatterns. Check the code quality and produce a report to the user with mandatory (if any) and recommended actions.

13. Python Execution Safety: The use of "python -c" or any form of inline Python execution is strictly forbidden. All Python logic must be contained within script files (e.g., in the tests/ directory or as temporary temp_*.py files). This ensures that all executions use the project's virtual environment and are traceable within the project's structure.

14. External File Safety: before modifying any file outside the project workspace or any file not protected by Git versioning, create an explicit backup/snapshot first. Never perform in-place edits on external Excel/templates or other user assets unless a recoverable backup has already been created and its location is known.

15. Root Cause and Simplicity Gate: before proposing or implementing bug fixes, reworks, behavior changes, or fixes to previous failed attempts, apply the root-cause-and-simplicity-gate skill. Identify the symptom, evidence, proximate cause, architectural/root cause, correct ownership/lifecycle, complexity risks, rejected workarounds, simplest responsibility-correct strategy, and verification plan. Do not proceed to code changes before the strategy is clear and explicitly authorized by the user.
